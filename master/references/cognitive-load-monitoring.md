# 认知负荷监控

## 核心原理

**公理**：认知负荷有上限，超载时学习效率归零。

**推论**：
1. 必须实时监控学生的认知负荷，而非按固定间隔传递
2. 负荷信号来自多个维度：回答速度、回答质量、主动求助、答非所问
3. 超载时必须立即停止新概念，转入巩固模式

---

## 认知负荷的三种状态

| 状态 | 定义 | 信号 | 教学策略 |
|------|------|------|---------|
| **正常** | 学生能跟上节奏，理解新概念 | 回答及时、准确、有细节 | 继续传递新概念 |
| **高负荷** | 学生接近上限，开始吃力 | 回答变慢、质量下降、需要重复 | 放慢节奏，增加验证频率 |
| **超载** | 学生已超载，无法理解新概念 | 主动求助、答非所问、沉默 | 立即停止新概念，转入巩固模式 |

---

## 负荷信号检测

### 信号 1：回答速度

**基线测量**：
- 在入学流程的"认知负荷上限探测"中，记录学生的基线回答速度
- 例如：学生平均 30 秒回答一个问题

**实时监控**：
```python
最近 3 轮的平均回答时间 = (t1 + t2 + t3) / 3

if 平均回答时间 > 基线 * 1.5:
    信号 = "回答速度变慢"
    负荷状态 = "高负荷"
```

**注意**：
- 回答时间过短也可能是问题（学生没有思考，随便答）
- 需要结合回答质量判断

### 信号 2：回答质量

**质量维度**：
1. **准确性**：回答是否正确
2. **完整性**：回答是否包含关键要素
3. **细节性**：回答是否有具体例子或推导

**评分标准**：
```python
def 评估回答质量(回答):
    分数 = 0
    
    if 回答正确:
        分数 += 0.4
    
    if 回答包含关键要素:
        分数 += 0.3
    
    if 回答有具体例子或推导:
        分数 += 0.3
    
    return 分数  # 0-1 之间

最近 3 轮的平均质量 = (q1 + q2 + q3) / 3

if 平均质量 < 0.6:
    信号 = "回答质量下降"
    负荷状态 = "高负荷"
```

### 信号 3：主动求助

**触发词**：
- "慢一点"
- "没懂"
- "能再解释一遍吗"
- "这个太难了"
- "我有点跟不上"

```python
if 学生说了触发词:
    信号 = "主动求助"
    负荷状态 = "超载"
```

### 信号 4：答非所问

**检测方法**：
```python
def 检测答非所问(问题, 回答):
    # 简化版：检查回答是否包含问题的关键词
    问题关键词 = 提取关键词(问题)
    回答关键词 = 提取关键词(回答)
    
    重叠度 = len(问题关键词 & 回答关键词) / len(问题关键词)
    
    if 重叠度 < 0.3:
        return True  # 答非所问
    else:
        return False

if 检测答非所问(问题, 回答):
    信号 = "答非所问"
    负荷状态 = "超载"
```

### 信号 5：沉默

**检测方法**：
```python
if 学生超过 2 分钟未回复:
    信号 = "沉默"
    负荷状态 = "超载"
```

## 负荷估算算法

### 综合评估

```python
def 估算当前负荷():
    信号列表 = []
    
    # 检测各种信号
    if 回答速度 > 基线 * 1.5:
        信号列表.append("回答速度变慢")
    
    if 回答质量 < 0.6:
        信号列表.append("回答质量下降")
    
    if 学生主动求助:
        信号列表.append("主动求助")
    
    if 答非所问:
        信号列表.append("答非所问")
    
    if 沉默:
        信号列表.append("沉默")
    
    # 根据信号数量判断负荷状态
    if "主动求助" in 信号列表 or "答非所问" in 信号列表 or "沉默" in 信号列表:
        return "超载", 信号列表
    
    elif len(信号列表) >= 2:
        return "高负荷", 信号列表
    
    else:
        return "正常", 信号列表
```

## 负荷应对策略

### 策略 1：正常负荷 → 继续传递

```python
if 负荷状态 == "正常":
    # 继续按计划传递下一个概念
    下一个概念 = 从依赖图中选择()
    传递概念(下一个概念)
```

### 策略 2：高负荷 → 放慢节奏

```python
if 负荷状态 == "高负荷":
    # 1. 放慢节奏：拆分当前概念
    if 当前概念.复杂度 >= 4:
        子概念列表 = 拆分概念(当前概念)
        传递概念(子概念列表[0])
    
    # 2. 增加验证频率
    验证频率 = "每个概念后验证"
    
    # 3. 增加例子数量
    例子数量 += 1
    
    # 4. 记录到 progress.md
    记录("检测到高负荷，已放慢节奏")
```

### 策略 3：超载 → 巩固模式

```python
if 负荷状态 == "超载":
    # 立即停止新概念
    停止传递新概念()
    
    # 转入巩固模式
    执行巩固模式()
    
    # 记录到 progress.md
    记录(f"检测到超载（信号：{信号列表}），已转入巩固模式")
```

## 巩固模式

### 目标
清空学生的认知负荷，巩固已学内容。

### 流程

**步骤 1：小费曼**
```
"我们先暂停一下，把刚才学的内容用你自己的话说一遍。"

听学生复述

如果复述卡壳：
    "卡在哪里了？我们一起回顾一下。"
```

**步骤 2：即时串联**
```
"我们刚学了 A、B、C 三个概念，你能说说它们之间的关系吗？"

听学生回答

补充："对，而且你注意到没有，它们都是在解决同一个问题：……"
```

**步骤 3：清理理解债务**
```
从 progress.md 中读取"理解债务"列表（之前验证失败但暂时跳过的概念）

选择 1-2 个债务概念，重新验证：
    "之前我们快速过了 X 这个概念，现在你能解释一下吗？"

如果仍然失败：
    重新讲解该概念
```

**步骤 4：休息提示**
```
"我们已经学了挺多内容，要不要休息一下？"

如果学生同意：
    更新 progress.md，记录"下次从何开始"
    结束本轮对话

如果学生拒绝：
    "好的，那我们继续。不过我会放慢一点节奏。"
    降低配置的 speed（临时）
```

## 动态配置调整

### 触发条件

每 5 轮检查一次：
```python
if 轮数 % 5 == 0:
    实际吞吐率 = 最近 5 轮学习的概念数 / 5
    配置吞吐率 = 配置.speed 对应的期望吞吐率
    
    if 实际吞吐率 < 配置吞吐率 * 0.7:
        # 学生实际负荷过高
        建议降速()
    
    elif 实际吞吐率 > 配置吞吐率 * 1.3:
        # 学生有余力
        建议提速()
```

### 降速流程

```python
def 建议降速():
    当前 speed = 配置.speed
    
    if 当前 speed == "sprint":
        建议 speed = "standard"
    elif 当前 speed == "standard":
        建议 speed = "deep"
    else:
        # 已经是 deep，无法再降
        建议 = "拆分概念"
    
    向学生说明：
        "我注意到你在最近几个概念上需要更多时间消化。
        要不要试试 {建议 speed} 模式？我会放慢节奏，给你更多时间理解。"
    
    if 学生同意:
        更新配置(speed = 建议 speed)
        在 progress.md 记录："检测到认知负荷过高，已降速到 {建议 speed}"
```

### 提速流程

```python
def 建议提速():
    当前 speed = 配置.speed
    
    if 当前 speed == "deep":
        建议 speed = "standard"
    elif 当前 speed == "standard":
        建议 speed = "sprint"
    else:
        # 已经是 sprint，无法再提
        return
    
    向学生说明：
        "我发现你学得比预期快，要不要试试 {建议 speed} 模式？
        我们可以加快节奏，覆盖更多内容。"
    
    if 学生同意:
        更新配置(speed = 建议 speed)
        在 progress.md 记录："检测到学习余力，已提速到 {建议 speed}"
```

## 常见陷阱

### 陷阱 1：忽略负荷信号

**错误做法**：
```
学生："这个太难了，能再解释一遍吗？"
导师："好的，我们继续下一个概念。"
→ 学生已经超载，但导师继续传递新概念
```

**正确做法**：
```
学生："这个太难了，能再解释一遍吗？"
导师："我们先暂停一下，把刚才学的内容用你自己的话说一遍。"
→ 转入巩固模式
```

### 陷阱 2：按固定间隔验证

**错误做法**：
```
导师讲了 5 个概念，然后验证
→ 学生在第 3 个概念就超载了，但导师没发现
```

**正确做法**：
```
导师讲完第 3 个概念（高复杂度），立即验证
→ 发现学生超载，转入巩固模式
```

### 陷阱 3：配置静态不变

**错误做法**：
```
学生选了 sprint 模式，但实际吞吐率只有 0.5
导师仍然按 sprint 模式执行
→ 学生持续超载
```

**正确做法**：
```
导师每 5 轮检查一次实际吞吐率
发现实际吞吐率 < 期望吞吐率 * 0.7
建议降速到 standard 模式
```

## 与 2.x 版本的区别

| 维度 | 2.x | 3.0 |
|------|-----|-----|
| 负荷监控 | 无实时监控 | 多维度信号检测 |
| 应对策略 | 按固定间隔验证 | 根据负荷状态动态调整 |
| 配置调整 | 配置静态 | 每 5 轮检查，动态调整 |
| 巩固模式 | 无明确定义 | 小费曼 + 即时串联 + 清理债务 |
| 记录机制 | 无 | 详细记录负荷日志 |
